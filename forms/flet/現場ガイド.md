# FormRescue (Flet) — 現場での実行と修正

このアプリは、ネイティブアプリにビルドせず、**Python で直接実行**して使う。
そのほうが、現場での修正が速いため。`.py` を直して、もう一度走らせるだけで
反映される(ビルド・署名・再配布が要らない)。

準備と依存は README「[開発環境](README.md#開発環境)」を前提とする。この資料は、
そこから先の「走らせて、直しながら使う」流れをまとめたもの。

## なぜ Python 直接実行か

- **修正が一瞬**: `.py` を編集 → 再実行。ビルド不要
- **AI と直せる**: 対象の `.py` を AI に渡して直させ、そのまま走らせる
- **道具を増やさない**: Xcode・署名・notarization・App Store が要らない

代わりに、動かす機械に Python + 依存が要る。これは一度の準備で済む。
見知らぬ非技術者に配るならネイティブビルドのほうが向くが、自分・現場の
設定済みの機械なら、こちらが速い。

## 前提

- Python 3(3.10 以上を推奨)。インストールは公式「Pythonのセットアップと利用」
  (https://docs.python.org/ja/3/using/index.html)の自分のOSの章に従う。
  分からなければ、自分のOSと状況を添えて AI に聞く
- ターミナル(macOS: ターミナル.app / Windows: PowerShell)

## 準備(初回のみ)

`flet/` フォルダで、仮想環境を作り、依存を入れる:

```sh
python3 -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -e .                 # または pip install flet httpx pyzbar Pillow
```

以降の作業は、すべてこの仮想環境の中で行う。ターミナルを開き直したら、
`source .venv/bin/activate` を再度実行する(プロンプトの先頭に `(.venv)` が
出ていれば有効)。

## 実行

```sh
python3 main.py     # flet run でも可
```

初回は README「[初回設定](README.md#初回設定)」に従い、Worker URL と PULL_TOKEN を
設定する。デスクトップでは QR カメラは使わず、手入力欄に、`deploy.py` が出力した
URL / TOKEN を入れる。設定は `data/inbox.db` に保存され、次回以降は不要
(README「[データの保存場所](README.md#データの保存場所)」)。

## 更新

新しいコードに入れ替えて、走らせ直すだけ:

```sh
git pull            # または、新しい .py ファイルをコピーで上書き
python3 main.py
```

データ(`data/inbox.db`)は OS のアプリドキュメント領域にあり、コードの
入れ替えでは消えない。バックアップも同領域に日次で残る。

## 現場で直す

Python 直接実行にしているのは、ここが目的:

1. 直したいところの `.py` を開く
   - 一覧の表示 → [list_view.py](list_view.py)
   - 個別の詳細 → [detail_view.py](detail_view.py)
   - URL / TOKEN の設定画面 → [settings_view.py](settings_view.py)
2. 直す。分からなければ、その `.py` を AI に渡して「こう直して」と頼む
3. `python3 main.py` で走らせて、確認する
4. 良ければ、そのまま使う(ビルドも再配布も要らない)

直し方は README「[実装方針](README.md#実装方針仕様書より)」に沿わせる:
状態管理ライブラリを増やさない、SQLite は raw SQL、削除の引き金は「確認」
操作のみ。この方針から外れる変更を AI にさせるときは、方針を渡したうえで頼む。

## 困ったとき

- **`python3` が無い / 古い** → 「前提」の Python インストールへ
- **`pip install` が失敗する** → `.venv` を有効化しているか確認
  (プロンプトに `(.venv)` が出ているか)。出ていなければ activate を実行
- **起動するが、受信箱が空** → Worker URL / PULL_TOKEN が正しいか。
  `curl "<worker>/items?after=0&limit=500" -H "Authorization: Bearer <TOKEN>"`
  で `{"items":[]}` が返れば、接続は正常(トークン違いなら 401)
- それ以外 → エラーの全文と、自分の OS を添えて、AI に聞く
